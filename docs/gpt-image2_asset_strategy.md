## 목적

GPT Image 2를 조선헌터스의 컨셉 디자인과 실제 게임 에셋 제작에 어디까지
활용할 수 있는지 현실적인 기준을 세운다.

판단 기준은 단순히 이미지를 생성할 수 있는지가 아니다.

- 반복 제작 시 캐릭터와 화풍이 유지되는가
- 게임 규격에 맞게 정렬하고 수정할 수 있는가
- 후처리 비용이 직접 제작보다 낮은가
- 런타임 에셋으로 안정적으로 변환할 수 있는가

GPT Image 2는 기본적으로 **완성품 생성기보다 고품질 원화 및 소재
생성기**로 활용한다.

## 적극 활용할 영역

### 컨셉아트와 캐릭터 시트

현재 도호, 귀새, 청연의 마스터 시트처럼 다음 제작물에 적극 사용한다.

- 캐릭터 정면, 측면, 후면 턴어라운드
- 의상, 장비, 재질 상세
- 표정 및 포즈 시트
- 요괴와 보스 디자인
- 지역, 던전, 건축물 분위기 원화
- 캐릭터 스킨과 장비 단계별 변형

공식 마스터 시트를 기준 이미지로 사용하고, 캐릭터별 고정 요소와 변경 가능
요소를 프롬프트에 분리해서 기록한다.

### 이벤트 컷신과 프로모션 이미지

정적인 서사 장면은 GPT Image 2의 강점을 가장 잘 활용할 수 있다.

- 메인 스토리 삽화
- 보스 등장 컷
- 지역 진입 컷
- 엔딩 및 분기 장면
- 로딩 화면
- 스토어 캡슐과 SNS 홍보물

텍스트는 이미지에 직접 생성하지 않고 코드나 편집 도구에서 합성한다. 동일
사건을 여러 장면으로 만들 때는 인물의 얼굴, 의상, 장비를 기준 이미지로
고정하고 한 번에 하나의 장면 변화만 요청한다.

### 초상화

캐릭터 시트를 기반으로 다음 변형을 제작한다.

- 기본, 기쁨, 분노, 슬픔, 부상 표정
- 인간, 각성, 타락 상태
- 대화창용 반신 및 얼굴 크롭
- NPC 등급별 초상화

초상화의 프레임, 배경, 등급 표시는 별도 UI 레이어로 관리한다.

## 아이콘

아이템 및 스킬 아이콘은 비용 대비 활용도가 가장 높은 실전 영역이다.

### 아이템 아이콘

적합한 항목:

- 무기, 방어구, 장신구
- 부적, 약재, 물약
- 요괴 전리품과 제작 재료
- 퀘스트 물품
- 강화, 파손, 저주 상태 변형

제작 규칙:

1. 아이콘을 한 장씩 개별 생성한다.
2. 정면 또는 3/4 시점을 카테고리별로 고정한다.
3. 단색 크로마키 배경과 넓은 여백을 사용한다.
4. 배경 제거 후 동일한 캔버스와 점유율로 정렬한다.
5. 최종 크기에서 외곽선과 명암을 수작업으로 보정한다.
6. 등급 프레임, 수량, 강화 수치는 UI에서 합성한다.

여러 아이콘이 들어간 한 장짜리 시트 생성은 탐색용으로만 사용한다. 개별
아이콘의 크기, 여백, 광원과 투명 배경을 통제하기 어렵기 때문이다.

### 스킬 아이콘

캐릭터마다 고유한 시각 언어를 고정한다.

| 캐릭터 | 핵심 모티프 | 주조색 |
|---|---|---|
| 도호 | 검기, 부적, 봉인진, 술법 문양 | 먹빛, 청색 |
| 귀새 | 가면, 붉은 리본, 곡도, 그림자 | 흑색, 적색 |
| 청연 | 부채, 방울, 봉황, 신령 | 상아색, 옥색 |
| 구미호 | 달, 꼬리, 여우불, 혈흔 | 백색, 적색 |

GPT Image 2는 중앙 심벌과 발광 효과를 생성한다. 공통 프레임, 레벨,
쿨다운, 잠금 상태는 게임 UI에서 처리한다.

## 픽셀아트

GPT Image 2는 픽셀아트처럼 보이는 이미지를 잘 만들지만, 규격화된 게임
스프라이트를 그대로 생산하는 데에는 한계가 있다.

주요 위험:

- 픽셀 크기와 외곽선 두께가 일정하지 않음
- 제한 팔레트를 정확히 유지하지 못함
- 프레임마다 신체 비율과 장비 형태가 달라짐
- 발 위치와 캐릭터 중심이 흔들림
- 방향 전환 시 무기 위치와 손잡이가 바뀜
- 스프라이트 시트의 셀 경계를 정확히 지키지 못함

권장 파이프라인:

1. 마스터 시트를 기준으로 고해상도 정적 스프라이트 원화를 만든다.
2. 전후좌우 방향을 각각 별도 이미지로 생성한다.
3. 목표 캔버스에 맞게 축소한다.
4. 팔레트 제한, 외곽선 정리, 발 위치 정렬을 수행한다.
5. Aseprite에서 최종 픽셀을 보정한다.
6. 애니메이션은 Pixellab 또는 수작업으로 제작한다.

GPT Image 2의 역할은 **캐릭터 디자인을 픽셀 언어로 번역하는 베이스
제작**까지로 제한한다. 연속 애니메이션 완제품은 맡기지 않는다.

## 맵 오브젝트

독립된 오브젝트는 활용 가능성이 높다.

적합한 항목:

- 장승, 석등, 우물, 제단, 봉인석
- 무덤, 비석, 항아리, 상자, 제기
- 등불, 창호, 기와 장식
- 나무, 바위, 꽃, 시체
- 요괴 둥지와 오염된 주술 오브젝트

제한이 큰 항목:

- 정확히 이어지는 지형 타일
- 완전한 seamless texture
- 모듈식 벽, 문, 지붕 세트
- 고정된 아이소메트릭 각도와 점유 면적
- 충돌 영역까지 고려한 구조물

독립 오브젝트는 GPT Image 2로 만들고, 길, 절벽, 벽, 지붕 같은 구조
타일은 전용 타일 제작 방식으로 만든다.

오브젝트 생성 시 다음 기준을 고정한다.

- 게임 맵 스크린샷을 시점 및 광원 참조로 제공
- 카메라 각도와 바닥 접점 명시
- 오브젝트의 타일 점유 크기 지정
- 그림자 방향과 길이 통일
- 크로마키 제거 후 바닥 앵커 정렬

## UI

GPT Image 2는 UI를 직접 완성하기보다 장식 소재와 전체 분위기 탐색에
사용한다.

적합한 작업:

- 한지, 먹, 목재, 청동 소재
- 창 테두리와 코너 장식
- 버튼과 슬롯 프레임
- 미니맵 및 초상화 테두리
- 퀘스트와 등급 문양
- 로딩 화면과 UI 목업

코드 또는 디자인 도구로 처리할 작업:

- 실제 화면 레이아웃
- 한글 텍스트와 수치
- 게이지, 버튼 상태, 포커스 상태
- 정렬, 간격, 반응형 배치
- 9-slice 분할과 런타임 조립

완성 UI 스크린샷 생성은 방향 탐색에만 사용한다. 최종 UI는 생성된 장식
부품을 코드로 조립한다.

## 투명 배경과 후처리

프로젝트용 독립 에셋은 단색 크로마키 배경으로 생성한 뒤 로컬에서 투명
배경으로 변환한다.

- 피사체에 없는 키 색상을 선택한다.
- 배경에는 그림자, 그라데이션, 바닥면, 반사를 넣지 않는다.
- 피사체 가장자리에 충분한 여백을 둔다.
- 투명화 후 알파 채널, 가장자리 번짐, 피사체 손실을 검수한다.
- 머리카락, 연기, 유리처럼 복잡한 반투명 요소는 별도 보정 비용을
  예상한다.

## 에셋 관리 구조

생성본과 게임 확정본을 구분한다.

```text
generated/
├── raw/          # GPT Image 2 원본과 프롬프트
└── rejected/     # 폐기하지 않고 비교용으로 보관할 생성본

processed/        # 배경 제거, 크롭, 색상 및 픽셀 보정

runtime/          # 게임에서 직접 사용하는 확정 에셋
```

각 에셋에는 가능한 한 다음 정보를 함께 기록한다.

- 캐릭터 또는 카테고리
- 에셋 종류
- 버전
- 생성 도구와 모델
- 최종 프롬프트
- 사용한 기준 이미지
- 후처리 내용
- 승인 상태

## 도입 우선순위

1. **아이템 아이콘**: 즉시 생산 도입
2. **스킬 아이콘**: 즉시 생산 도입
3. **이벤트 컷신과 초상화**: 적극 도입
4. **독립 맵 오브젝트**: 규격 테스트 후 조건부 도입
5. **정적 캐릭터 픽셀 베이스**: 하이브리드 방식
6. **UI 장식 소재**: 생성 후 코드 조립
7. **타일셋과 애니메이션**: 보조 및 참고 용도

## 첫 벤치마크

다음 네 묶음을 동일한 기준으로 제작하여 실제 생산성을 측정한다.

1. 64px 아이템 아이콘 6종
2. 64px 스킬 아이콘 6종
3. 탑다운 맵 오브젝트 4종
4. 도호 정적 128px 스프라이트 4방향

각 테스트에서 다음 값을 기록한다.

- 최초 생성 성공률
- 캐릭터 및 화풍 일치도
- 배경 제거 성공 여부
- 최종 규격까지 걸린 후처리 시간
- 수작업 대비 절감 시간
- 런타임 채택 여부

벤치마크 결과를 기준으로 카테고리별 프롬프트 템플릿과 승인 규칙을
확정한다.

## 최종 원칙

- GPT Image 2는 컨셉, 컷신, 초상화, 아이콘에 적극 사용한다.
- 맵 오브젝트는 독립형 소품 위주로 사용한다.
- 픽셀 캐릭터는 생성과 수작업을 결합한다.
- UI는 장식 소재만 생성하고 구조와 텍스트는 코드로 만든다.
- 타일셋과 애니메이션은 현재 단계에서 최종 생성물로 기대하지 않는다.
- 생성 원본, 후처리본, 런타임 확정본을 분리해 관리한다.
